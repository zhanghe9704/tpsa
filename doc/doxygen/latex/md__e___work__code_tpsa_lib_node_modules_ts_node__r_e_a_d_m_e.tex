\end{DoxyItemize}\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md2993}{}\doxysection{Overview}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md2993}
ts-\/node is a Type\+Script execution engine and REPL for Node.\+js.

It JIT transforms Type\+Script into Java\+Script, enabling you to directly execute Type\+Script on Node.\+js without precompiling. This is accomplished by hooking node\textquotesingle{}s module loading APIs, enabling it to be used seamlessly alongside other Node.\+js tools and libraries.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md2994}{}\doxysubsection{Features}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md2994}

\begin{DoxyItemize}
\item Automatic sourcemaps in stack traces
\item Automatic {\ttfamily tsconfig.\+json} parsing
\item Automatic defaults to match your node version
\item Typechecking (optional)
\item REPL
\item Write standalone scripts
\item Native ESM loader
\item Use third-\/party transpilers
\item Use custom transformers
\item Integrate with test runners, debuggers, and CLI tools
\item Compatible with pre-\/compilation for production
\end{DoxyItemize}

\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3003}{}\doxysubsection{Via tsconfig.\+json (recommended)}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3003}
ts-\/node automatically finds and loads {\ttfamily tsconfig.\+json}. Most ts-\/node options can be specified in a {\ttfamily \char`\"{}ts-\/node\char`\"{}} object using their programmatic, camel\+Case names. We recommend this because it works even when you cannot pass CLI flags, such as {\ttfamily node -\/-\/require ts-\/node/register} and when using shebangs.

Use {\ttfamily -\/-\/skip\+Project} to skip loading the {\ttfamily tsconfig.\+json}. Use {\ttfamily -\/-\/project} to explicitly specify the path to a {\ttfamily tsconfig.\+json}.

When searching, it is resolved using \href{https://www.typescriptlang.org/docs/handbook/tsconfig-json.html}{\texttt{ the same search behavior as {\ttfamily tsc}}}. By default, this search is performed relative to the entrypoint script. In {\ttfamily -\/-\/cwd\+Mode} or if no entrypoint is specified -- for example when using the REPL -- the search is performed relative to {\ttfamily -\/-\/cwd} / {\ttfamily process.\+cwd()}.

You can use this sample configuration as a starting point\+:

\DoxyCodeLine{\{}
\DoxyCodeLine{  // This is an alias to @tsconfig/node16: https://github.com/tsconfig/bases}

\DoxyCodeLine{}
\DoxyCodeLine{  // Most ts-\/node options can be specified here using their programmatic names.}
\DoxyCodeLine{  "{}ts-\/node"{}: \{}
\DoxyCodeLine{    // It is faster to skip typechecking.}
\DoxyCodeLine{    // Remove if you want ts-\/node to do typechecking.}
\DoxyCodeLine{    "{}transpileOnly"{}: true,}
\DoxyCodeLine{}
\DoxyCodeLine{    "{}files"{}: true,}
\DoxyCodeLine{}
\DoxyCodeLine{    "{}compilerOptions"{}: \{}
\DoxyCodeLine{      // compilerOptions specified here will override those declared below,}
\DoxyCodeLine{      // but *only* in ts-\/node.  Useful if you want ts-\/node and tsc to use}
\DoxyCodeLine{      // different options with a single tsconfig.json.}
\DoxyCodeLine{    \}}
\DoxyCodeLine{  \},}
\DoxyCodeLine{  "{}compilerOptions"{}: \{}
\DoxyCodeLine{    // typescript options here}
\DoxyCodeLine{  \}}
\DoxyCodeLine{\}}

\end{DoxyCode}


Our bundled \href{https://unpkg.com/browse/ts-node@latest/tsconfig.schema.json}{\texttt{ JSON schema}} lists all compatible options.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3004}{}\doxysubsubsection{@tsconfig/bases}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3004}

\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3005}{}\doxysubsubsection{Default config}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3005}
If no {\ttfamily tsconfig.\+json} is loaded from disk, ts-\/node will use the newest recommended defaults from \href{https://github.com/tsconfig/bases/}{\texttt{ @tsconfig/bases}} compatible with your {\ttfamily node} and {\ttfamily typescript} versions. With the latest {\ttfamily node} and {\ttfamily typescript}, this is \href{https://github.com/tsconfig/bases/blob/master/bases/node16.json}{\texttt{ {\ttfamily @tsconfig/node16}}}.

Older versions of {\ttfamily typescript} are incompatible with {\ttfamily @tsconfig/node16}. In those cases we will use an older default configuration.

When in doubt, {\ttfamily ts-\/node -\/-\/show\+Config} will log the configuration being used, and {\ttfamily ts-\/node -\/vv} will log {\ttfamily node} and {\ttfamily typescript} versions.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3006}{}\doxysubsection{$<$tt$>$node$<$/tt$>$ flags}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3006}

\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3007}{}\doxysection{Options}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3007}
All command-\/line flags support both {\ttfamily -\/-\/camel\+Case} and {\ttfamily -\/-\/hyphen-\/case}.

Most options can be declared in your tsconfig.\+json\+: \href{\#via-tsconfigjson-recommended}{\texttt{ Configuration via tsconfig.\+json}}

\end{DoxyCode}

\begin{DoxyCode}{0}
\DoxyCodeLine{const a = foo?.bar;}

\end{DoxyCode}

ts-\/node does {\itshape not} eagerly load {\ttfamily files}, {\ttfamily include} or {\ttfamily exclude} by default. This is because a large majority of projects do not use all of the files in a project directory (e.\+g. {\ttfamily Gulpfile.\+ts}, runtime vs tests) and parsing every file for types slows startup time. Instead, ts-\/node starts with the script file (e.\+g. {\ttfamily ts-\/node index.\+ts}) and Type\+Script resolves dependencies based on imports and references.

Occasionally, this optimization leads to missing types. Fortunately, there are other ways to include them in typechecking.

For global definitions, you can use the {\ttfamily type\+Roots} compiler option. This requires that your type definitions be structured as type packages (not loose Type\+Script definition files). More details on how this works can be found in the \href{https://www.typescriptlang.org/docs/handbook/tsconfig-json.html\#types-typeroots-and-types}{\texttt{ Type\+Script Handbook}}.

Example {\ttfamily tsconfig.\+json}\+:

\DoxyCodeLine{    "{}typeRoots"{} : ["{}./node\_modules/@types"{}, "{}./typings"{}]}
\DoxyCodeLine{  \}}
\DoxyCodeLine{\}}

\end{DoxyCode}

\begin{DoxyCode}{0}
\DoxyCodeLine{<project\_root>/}
\DoxyCodeLine{-\/-\/ tsconfig.json}
\DoxyCodeLine{-\/-\/ typings/}
\DoxyCodeLine{  -\/-\/ <module\_name>/}
\DoxyCodeLine{    -\/-\/ index.d.ts}

\end{DoxyCode}

\DoxyCodeLine{    // module definitions go here}
\DoxyCodeLine{\}}

\end{DoxyCode}

This may be helpful if you prefer not to change your {\ttfamily compiler\+Options} or structure your type definitions for {\ttfamily type\+Roots}. Below is an example of a triple-\/slash directive as a relative path within your project\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{/// <reference path="{}./types/lib\_greeter"{} />}

\DoxyCodeLine{const g = new Greeter();}
\DoxyCodeLine{g.sayHello();}

If none of the above work, and you {\itshape must} use {\ttfamily files}, {\ttfamily include}, or {\ttfamily exclude}, enable our \href{\#files}{\texttt{ {\ttfamily files}}} option.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3063}{}\doxysubsection{npx, yarn dlx, and node\+\_\+modules}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3063}
When executing Type\+Script with {\ttfamily npx} or {\ttfamily yarn dlx}, the code resides within a temporary {\ttfamily node\+\_\+modules} directory.

The contents of {\ttfamily node\+\_\+modules} are ignored by default. If execution fails, enable \href{\#skipignore}{\texttt{ {\ttfamily skip\+Ignore}}}.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3064}{}\doxysection{Performance}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3064}
These tricks will make ts-\/node faster.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3065}{}\doxysubsection{Skip typechecking}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3065}
It is often better to typecheck as part of your tests or linting. You can run {\ttfamily tsc -\/-\/no\+Emit} to do this. In these cases, ts-\/node can skip typechecking, making it much faster.

To skip typechecking in ts-\/node, do one of the following\+:


\begin{DoxyItemize}
\item Enable \href{\#swc}{\texttt{ swc}}
\begin{DoxyItemize}
\item This is by far the fastest option

\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3068}{}\doxysubsection{How it works}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3068}
ts-\/node works by registering hooks for {\ttfamily .ts}, {\ttfamily .tsx}, {\ttfamily .js}, and/or {\ttfamily .jsx} extensions.

Vanilla {\ttfamily node} loads {\ttfamily .js} by reading code from disk and executing it. Our hook runs in the middle, transforming code from Type\+Script to Java\+Script and passing the result to {\ttfamily node} for execution. This transformation will respect your {\ttfamily tsconfig.\+json} as if you had compiled via {\ttfamily tsc}.

We also register a few other hooks to apply sourcemaps to stack traces and remap from {\ttfamily .js} imports to {\ttfamily .ts}.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3069}{}\doxysubsection{Ignored files}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3069}
ts-\/node transforms certain files and ignores others. We refer to this mechanism as \char`\"{}scoping.\char`\"{} There are various options to configure scoping, so that ts-\/node transforms only the files in your project.

\begin{quote}
{\bfseries{Warning\+:}}

An ignored file can still be executed by node.\+js. Ignoring a file means we do not transform it from Type\+Script into Java\+Script, but it does not prevent execution.

If a file requires transformation but is ignored, node may either fail to resolve it or attempt to execute it as vanilla Java\+Script. This may cause syntax errors or other failures, because node does not understand Type\+Script type syntax nor bleeding-\/edge ECMAScript features. \end{quote}
\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3070}{}\doxysubsubsection{File extensions}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3070}
{\ttfamily .js} and {\ttfamily .jsx} are only transformed when \href{https://www.typescriptlang.org/docs/handbook/compiler-options.html\#compiler-options}{\texttt{ {\ttfamily allow\+Js}}} is enabled.

{\ttfamily .tsx} and {\ttfamily .jsx} are only transformed when \href{https://www.typescriptlang.org/docs/handbook/jsx.html}{\texttt{ {\ttfamily jsx}}} is enabled.

\begin{quote}
{\bfseries{Warning\+:}}

When ts-\/node is used with {\ttfamily allow\+Js}, {\itshape all} non-\/ignored Java\+Script files are transformed by ts-\/node. \end{quote}
\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3071}{}\doxysubsubsection{Skipping $<$tt$>$node\+\_\+modules$<$/tt$>$}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3071}
By default, ts-\/node avoids compiling files in {\ttfamily /node\+\_\+modules/} for three reasons\+:


\begin{DoxyEnumerate}
\item Modules should always be published in a format node.\+js can consume
\item Transpiling the entire dependency tree will make your project slower
\item Differing behaviours between Type\+Script and node.\+js (e.\+g. ES2015 modules) can result in a project that works until you decide to support a feature natively from node.\+js
\end{DoxyEnumerate}

If you need to import uncompiled Type\+Script in {\ttfamily node\+\_\+modules}, use \href{\#skipignore}{\texttt{ {\ttfamily -\/-\/skip\+Ignore}}} or \href{\#skipignore}{\texttt{ {\ttfamily TS\+\_\+\+NODE\+\_\+\+SKIP\+\_\+\+IGNORE}}} to bypass this restriction.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3072}{}\doxysubsubsection{Skipping pre-\/compiled Type\+Script}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3072}

\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3076}{}\doxysubsubsection{Why is this not built-\/in to ts-\/node?}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3076}
The official Type\+Script Handbook explains the intended purpose for {\ttfamily \char`\"{}paths\char`\"{}} in \href{https://www.typescriptlang.org/docs/handbook/module-resolution.html\#additional-module-resolution-flags}{\texttt{ \char`\"{}\+Additional module resolution flags\char`\"{}}}.

\begin{quote}
The Type\+Script compiler has a set of additional flags to {\itshape inform} the compiler of transformations that are expected to happen to the sources to generate the final output.

It is important to note that the compiler will not perform any of these transformations; it just uses these pieces of information to guide the process of resolving a module import to its definition file. \end{quote}
This means {\ttfamily \char`\"{}paths\char`\"{}} are intended to describe mappings that the build tool or runtime {\itshape already} performs, not to tell the build tool or runtime how to resolve modules. In other words, they intend us to write our imports in a way {\ttfamily node} already understands. For this reason, ts-\/node does not modify {\ttfamily node}\textquotesingle{}s module resolution behavior to implement {\ttfamily \char`\"{}paths\char`\"{}} mappings.\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3077}{}\doxysubsection{Third-\/party compilers}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3077}

\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3078}{}\doxysubsection{Transpilers}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3078}
ts-\/node supports third-\/party transpilers as plugins. Transpilers such as swc can transform Type\+Script into Java\+Script much faster than the Type\+Script compiler. You will still benefit from ts-\/node\textquotesingle{}s automatic {\ttfamily tsconfig.\+json} discovery, sourcemap support, and global ts-\/node CLI. Plugins automatically derive an appropriate configuration from your existing {\ttfamily tsconfig.\+json} which simplifies project boilerplate.

\begin{quote}
{\bfseries{What is the difference between a compiler and a transpiler?}}

For our purposes, a compiler implements Type\+Script\textquotesingle{}s API and can perform typechecking. A third-\/party transpiler does not. Both transform Type\+Script into Java\+Script. \end{quote}
\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3079}{}\doxysubsubsection{Third-\/party plugins}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3079}

To write your own transpiler plugin, check our \href{https://typestrong.org/ts-node/api/interfaces/TranspilerModule.html}{\texttt{ API docs}}.

Plugins are {\ttfamily require()}d by ts-\/node, so they can be a local script or a node module published to npm. The module must export a {\ttfamily create} function described by our \href{https://typestrong.org/ts-node/api/interfaces/TranspilerModule.html}{\texttt{ {\ttfamily Transpiler\+Module}}} interface. {\ttfamily create} is invoked by ts-\/node at startup to create one or more transpiler instances. The instances are used to transform Type\+Script into Java\+Script.

For a working example, check out out our bundled swc plugin\+: \href{https://github.com/TypeStrong/ts-node/blob/main/src/transpilers/swc.ts}{\texttt{ https\+://github.\+com/\+Type\+Strong/ts-\/node/blob/main/src/transpilers/swc.\+ts}}\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3081}{}\doxysubsection{Module type overrides}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3081}
\begin{quote}
Wherever possible, it is recommended to use Type\+Script\textquotesingle{}s \href{https://www.typescriptlang.org/docs/handbook/esm-node.html}{\texttt{ {\ttfamily Node\+Next} or {\ttfamily Node16} mode}} instead of the options described in this section. Setting {\ttfamily \char`\"{}module\char`\"{}\+: \char`\"{}\+Node\+Next\char`\"{}} and using the {\ttfamily .cts} file extension should work well for most projects. \end{quote}
When deciding how a file should be compiled and executed -- as either Common\+JS or native ECMAScript module -- ts-\/node matches {\ttfamily node} and {\ttfamily tsc} behavior. This means Type\+Script files are transformed according to your {\ttfamily tsconfig.\+json} {\ttfamily \char`\"{}module\char`\"{}} option and executed according to node\textquotesingle{}s rules for the {\ttfamily package.\+json} {\ttfamily \char`\"{}type\char`\"{}} field. Set {\ttfamily \char`\"{}module\char`\"{}\+: \char`\"{}\+Node\+Next\char`\"{}} and everything should work.

In rare cases, you may need to override this behavior for some files. For example, some tools read a {\ttfamily name-\/of-\/tool.\+config.\+ts} and require that file to execute as Common\+JS. If you have {\ttfamily package.\+json} configured with {\ttfamily \char`\"{}type\char`\"{}\+: \char`\"{}module\char`\"{}} and {\ttfamily tsconfig.\+json} with {\ttfamily \char`\"{}module\char`\"{}\+: \char`\"{}esnext\char`\"{}}, the config is native ECMAScript by default and will raise an error. You will need to force the config and any supporting scripts to execute as Common\+JS.

In these situations, our {\ttfamily module\+Types} option can override certain files to be Common\+JS or ESM. Similar overriding is possible by using {\ttfamily .mts}, {\ttfamily .cts}, {\ttfamily .cjs} and {\ttfamily .mjs} file extensions. {\ttfamily module\+Types} achieves the same effect for {\ttfamily .ts} and {\ttfamily .js} files, and {\itshape also} overrides your {\ttfamily tsconfig.\+json} {\ttfamily \char`\"{}module\char`\"{}} config appropriately.

\end{DoxyItemize}\hypertarget{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3082}{}\doxysubsubsection{Caveats}\label{md__e___work__code_tpsa_lib_node_modules_ts_node__r_e_a_d_m_e_autotoc_md3082}
